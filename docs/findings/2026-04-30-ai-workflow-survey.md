## 代表的な運用パターン

**Spec-first 単発セッション型** — SoT は `spec.md`、OpenSpec、あるいは会話から起こした plan です。人間は仕様の明確化と受け入れ基準を担当し、AI は実装と試行錯誤を担当します。コンテキスト管理は「大きい課題を小さな実装単位に割る」「必要なら新しいセッションに切る」が中心で、モデル選択は明示的なチケット属性ではなく、タスクの重さでその場で決めることが多いです。計測は希薄で、強みは速さと軽さ、弱みは handoff や監査の弱さです。代表例は entity["people","Harper Reed","software engineer"] の `spec.md` + 小刻み plan、entity["people","Addy Osmani","web engineer"] の spec-first / tasks JSON、entity["company","HumanLayer","ai devtools"] の research-plan-implement です。個人開発や小規模 greenfield には非常に合いますが、複数 AI handoff にはそのままでは弱いです。citeturn19view5turn19view6turn19view7turn19view9turn41view0

**AGENTS/CLAUDE とセッション衛生型** — SoT は Issue ではなく、`AGENTS.md` / `CLAUDE.md` と session hygiene です。人間は「何を毎回読ませるか」を設計し、AI はそのガイドを前提に走ります。コンテキスト管理は `/clear`、手動 `/compact`、subagent での探索分離、階層的な AGENTS override が中心で、モデル選択は「親は強いモデル、探索サブエージェントは安いモデル」が典型です。計測は通常行われません。代表例は entity["people","Dawid Rycerz","software engineer"] の Opus 本体 + Haiku/Sonnet subagent、HumanLayer の parent=Opus / subagent=Sonnet or Haiku、OpenAI の Codex における階層 AGENTS と軽量 subagent routing、entity["company","Vercel","frontend platform"] の AGENTS.md 常時注入評価です。強みは context tax を抑えやすいこと、弱みは external state が薄く履歴監査が弱いことです。brownfield の個人〜少人数チームに向きます。citeturn19view0turn19view1turn19view2turn41view1turn41view2turn32view0turn32view2turn40view1

**Issue / PR / Jira control-plane 型** — SoT は GitHub Issue、Jira issue、または PR です。人間と AI の責務分離は、細かいラベル体系よりも、**assignee / issue assignment / branch / PR review / merge gate** で表現されることが多く、AI が誤って閉じる・マージすることを防ぐ主な仕組みは「AI は branch/PR まで、人間が review/merge」という分離です。コンテキストは issue 本文、custom instructions、repo memory、historic PR を必要範囲だけ読む形に寄っています。モデル選択は 2026 年時点では GitHub 上の model picker で session 時に選ぶ式がかなり強く、ticket metadata での model 表現は相対的に減っています。計測は PR lifecycle 指標として取りやすいです。代表例は GitHub Copilot cloud agent の issue/Jira 割当フローと、entity["people","Simon Willison","software developer"] が使う async coding agent の「PR を成果物として受け取る」運用です。小〜中規模チームの非同期開発に向きます。citeturn17view1turn17view2turn16view13turn17view8turn17view9turn38view2

**External tracker + orchestrator 型** — SoT は外部 tracker のまま維持しつつ、その上に dispatcher / state machine / workspace isolation を載せるやり方です。責務分離は最も明確で、tracker state、handoff state、terminal state、review comment handling、CI 連動で表現されます。コンテキストは ticket からレンダリングした prompt と repo guidance に絞られ、モデルは backend adapter や custom agent で切り替えます。計測も比較的進んでいます。代表例は Zenn の driller による Issue Workflow、Sortie、GitHub Agentic Workflows です。強みは durable handoff・監査・並列運用・コスト可視化、弱みは setup と運用設計の重さです。複数 AI、複数ブランチ、CI 連動が前提の serious project に向きます。citeturn20view3turn23view0turn27view0turn27view3turn27view4

**Agent-native tracker 型** — SoT を GitHub Issue から repo 内・ローカル DB 内の tracker に移し、「人間向け tracker」ではなく「AI が扱いやすい task graph」を作る流れです。責務分離は task graph、dependency、review agent、close/push 条件で表現されます。コンテキストは query ベースで必要分だけ引き、モデルは agent-agnostic にされがちです。計測はまだ発展途上ですが、context 最適化や conflict 回避の設計はかなり進んでいます。代表例は Beans、Beads、PlanDB と、entity["people","Sion Williams","software engineer"] の OpenCode + Beads 運用です。強みは multi-agent と長期タスクに強いこと、弱みは既存の GitHub/Jira エコシステム統合がまだ弱いことです。個人〜小規模でも agent 数が増える文脈に向きます。citeturn24view0turn24view1turn24view2turn24view3turn24view4turn24view5turn36view0turn36view2

**Parallel sessions / async swarm 型** — SoT は tracker ではなく、**thread / worktree / separate checkout / cloud task / PR artifact** に寄ります。責務分離は「別ディレクトリ・別 worktree で衝突を避ける」「危険な作業は async cloud agent に逃がす」「レビュー時だけ人が強く介入する」で実現されます。コンテキスト管理は formal handoff より isolation 重視で、モデルはツールごと・agent ごとに選ばれます。計測は普段は薄く、必要時に thread logs を見る程度です。代表例は Simon Willison の複数 terminal / fresh checkout 運用、Anthropic の data-infra team の複数インスタンス運用、Codex app / cloud の parallel threads です。強みはスピード、弱みは state の一元管理が弱いことです。低〜中リスクの並列作業や research tasks に向きます。citeturn38view1turn38view2turn28view0turn31view5turn33view0turn26view0

## 論点別の実態

**SoT と責務分離** については、個人・小規模では spec/AGENTS/セッションそのものが SoT になりやすく、業務・非同期・複数人になるほど Issue/Jira/PR に寄ります。責務分離の表現手段としては、公開例では **細密な label taxonomy より、assignee・issue assignment・isolated branch/worktree・PR review gate・permission mode・hooks** のほうが多く見つかりました。AI が close / merge / push を誤ることへの対策も、ラベル規約より **review 前提の PR 運用**、**safe-outputs**、**allow/deny list**、**hook での事前検査**、**auto mode の権限制御** に寄っています。citeturn17view1turn17view2turn20view3turn27view0turn35view0turn35view1turn30view0turn30view4

**複数 AI handoff とコンテキスト** については、「明示 handoff をする」よりも「handoff が要らないように分離する」発想が強くなっています。HumanLayer は frequent intentional compaction と subagent 分離を推し、Claude Code と Codex は subagent / worktree / thread を公式に整備し、Vercel は agent が毎ターン確実に読める AGENTS.md のほうが skills より安定すると報告しています。Simon Willison も、同一 repo 上で複数 agent が干渉しないように separate checkout を使っており、HN でも worktree 等の分離なしでは agent が互いの変更を踏むという anecdote が出ています。つまり、**「全 Issue を毎回丸ごと読ませる」より、「必要最小限の context packet を、worktree と subagent で隔離する」方向が 2026 年の実運用に近い** です。citeturn41view0turn41view1turn26view1turn26view0turn40view1turn38view1turn11search5

**モデル選択とコスト管理** は、公開事例では **ticket label ではなく session / subagent / UI picker / harness 設定** に寄っています。Dawid Rycerz は main flow に Opus、探索系 subtask に Haiku/Sonnet を使い分け、HumanLayer も parent に高性能モデル、subagent に安価モデルを当てています。Claude Code の built-in Explore は Haiku、Codex も lighter subagent 用に mini 系を推奨し、GitHub では Claude/Codex 用 model picker が Agents tab から使えるようになりました。要するに、**model routing は ticket metadata というより「実行面の設定」に下りてきている** ということです。なお、`claude-code-router` のような専用 proxy を「実運用で広く使っている」と断言できる公開事例は、今回の調査では十分に集まりませんでした。citeturn19view0turn41view1turn26view1turn31view8turn31view9turn17view6

**計測と失敗事例** は、まだ成熟途上です。個人ブログでは token accounting を厳密に出している例は少なく、計測が明示されるのは Sortie の cost tracking、GitHub の PR outcome metrics、gh-aw の audit report のように、**ツール側・基盤側の計測** が多いです。一方、失敗パターンはかなり収束しています。Markdown ベースの task 管理は drift しやすく、長い stale session は忘却や反復の原因になり、巨大な AI PR を人が毎回読み切るのは持続しません。こうした失敗が、Beads / PlanDB / Beans のような agent-native tracker、あるいは HumanLayer の compaction 重視、Claude Code の auto mode / subagent / worktree といった方向を後押ししています。citeturn23view0turn17view1turn27view4turn21view4turn38view0turn41view0turn24view2turn24view5

## あなたの運用との距離

あなたの現状に **最も近い** のは、**「external tracker + orchestrator」型**、単体の近似例としては driller の Issue Workflow です。GitHub Issue を仕事単位にし、Issue から branch を切り、進捗を Issue に報告し、並列時は `git worktree` で隔離し、レビュー対応の一部は人間が実行する、という構成は、あなたの「Issue を SoT にし、AI 間 handoff を tracker 上に残し、人間がクロージング権限を持つ」発想とかなり近いです。Sortie も同系統で、active / handoff / terminal の状態機械、isolated workspace、CI feedback、cost tracking まで揃っており、あなたの運用をさらに機械可読化した先にあるもの、と見てよいです。citeturn20view3turn23view0

あなたの現状に **最も遠い** のは、**「parallel sessions / session-local」型**、単体の近似例としては Simon Willison の並列 terminal 運用や、entity["company","Builder.io","visual dev platform"] の entity["people","Steve Sewell","builder ceo"] が書く Cursor の YOLO/test-first 運用です。ここでは operational unit が Issue ではなく、**その場の thread / directory / test loop** であり、handoff は comment schema ではなく fresh session / separate checkout / PR artifact によって吸収されます。個人の高速反復には向きますが、複数 AI が同一タスクを跨いで再開し続けるには、あなたの現状よりかなり薄いガバナンスです。citeturn38view1turn35view0

## 軽量化と重量化

**軽量化するなら最初に捨てやすいもの** は三つあります。第一に **`model:` ラベル** です。公開されている実運用では model choice は issue metadata より、session 時の picker、subagent ごとの固定モデル、あるいは harness 設定に下りています。第二に **長文の AI-to-AI handoff コメント** です。multi-agent 文脈で durable handoff 自体は重要ですが、自由記述の長文より、goal / current state / acceptance criteria / blocking point / links だけを残す「圧縮 handoff」にしたほうが 2026 年の流儀に合います。第三に **毎回のフル履歴再読** です。Issue 全体・全コメントを毎回読ませるより、Issue 本文 + 最新 plan + 未解決レビューだけを context packet として生成するほうが、いまの subagent / AGENTS / compaction 文化と整合します。逆に、**commit と issue の結び付け**、**human-only close**、**明示的な review gate** は、コストに対して守備力が高いので、削る優先度は低いです。citeturn19view0turn41view1turn17view6turn41view0turn40view1turn30view0turn30view4

**重量化するなら足すべきもの** は、自由記述を増やすことではなく、**状態遷移を機械可読化すること** です。具体的には、Sortie のような active / handoff / terminal state、gh-aw の safe-outputs と approval gate、driller の issue-number → worktree → review → merge の順序制御のように、runner が理解できる state machine を持つことです。次に、**run 単位のメトリクス** を足す価値が高いです。token、cost、retry 回数、CI 再実行回数、軽量モデル失敗→上位モデル再実行率、PR merge lead time まで取れると、governance が「設計思想」から「改善可能な運用」に変わります。さらに、Sion Williams の Beads 運用や driller の hachimoku のように、**実装 agent と review agent を分離** すると、複数 AI の handoff が「コメントの受け渡し」ではなく「役割間の品質ゲート」に変わります。citeturn23view0turn27view0turn20view3turn27view4turn17view1turn36view2

## 2026年のトレンドと時代遅れ判定

2026 年時点の大きな変化は、**以前は自前で盛る必要があった制御面が、プラットフォーム側に降りてきた** ことです。GitHub では Agent HQ / mission control、Claude/Codex の model picker、Jira 割当、issue assignment API、session logs、PR outcome metrics が揃い、Anthropic は auto mode で permission fatigue を吸収し、Codex app は built-in worktree と parallel threads と automations を持つようになりました。加えて AGENTS.md は cross-tool standard として整い、Vercel の eval では「skills を呼ぶより AGENTS.md を常時与える」ほうが安定でした。インフラ側でも、Vercel では 2026 年 4 月時点で weekly deployment の 30% 超が coding agent 起点になっています。つまり、**issue governance を捨てる方向ではなく、governance の一部が platform primitive 化した** と見るのが妥当です。citeturn16view12turn16view13turn16view14turn17view8turn17view1turn30view0turn33view0turn40view0turn40view1turn40view2

その前提で、**明らかに古くなりつつある** のは、**ローカル Markdown handoff を唯一の durable memory にすること**、**polyglot repo 全体を一枚の巨大な instruction file で管理すること**、**長いタスクで毎アクション手動承認を続けること**、**毎回フル履歴を読み直して context を膨らませること** です。Markdown handoff は task drift の不満から専用 tracker が生まれており、巨大 instruction file は context tax の副作用が強く、手動承認の連打は approval fatigue に陥りやすく、フル履歴再読は subagent / compaction / compressed AGENTS の潮流に逆行します。逆に、**Issue を SoT にすること自体は時代遅れではありません**。むしろ GitHub issue assignment、Jira integration、tracker-driven orchestrator が増えたことで、Issue/Jira は依然として有力な control plane の一つです。citeturn21view4turn24view2turn24view5turn36view2turn32view0turn30view0turn29view0turn41view0turn26view1

## 結論

あなたの仮説を、この調査の事例数に引きつけて言い切ると、**事実に近いのは A** です。公開事例の中心は、**軽量な spec/AGENTS/セッション運用** と **Issue/PR/Jira を control plane にした非同期運用** に分かれており、あなたの GitHub Issue lifecycle governance を「すでに時代遅れになった重厚長大な設計」とみなせる根拠は見当たりませんでした。むしろ、複数 AI handoff と durable state を必要とする文脈では、driller、Sortie、gh-aw、Beads/PlanDB 系の出現が示すように、**外部 tracker か agent-native tracker のどちらかで state を明示管理する方向** が強まっています。したがって、あなたの設計は「焦点が違うテンプレに載っていないだけ」という説明が、全体として最も近いです。citeturn20view3turn23view0turn27view0turn24view2turn24view5turn39search3turn39search4

ただし、**B が部分的に当たる点** も明確です。特に **model routing を issue label に持つこと**、**自由記述 handoff を長く残すこと**、**フル履歴を毎回読ませること** は、2026 年の platform primitives と context engineering の進歩で、より軽い代替に置き換えやすくなっています。要するに、**あなたの「Issue lifecycle governance の核」は妥当で、削るべきは核ではなく周辺の表現方法** です。これが今回集めた事例から見える、最も高信頼な結論です。citeturn17view6turn41view0turn40view1turn30view0turn19view0
